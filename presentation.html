01 · Problem Statement

Cities are sensor-rich,
insight-poor.

Traffic, air quality, power, water, and noise telemetry all exist — but each lives in its own dashboard, in its own department, on its own clock.
  • Operators can't see one coherent picture of a neighborhood's health.
  • Failures in one system cascade invisibly into adjacent districts and other utilities.
  • By the time an incident is obvious on a single meter, the ripple has already happened.
02 · Why We Need This

A shared real-time view of urban health.

Composite, not siloed

One Urban Health Score (UHS) per district — a weighted blend of traffic, air, power, water, and noise — so operators and residents see one number that actually moves when things go wrong.

Ripple-aware

Real infrastructure failures don't respect district lines. When one neighborhood's score collapses, adjacent areas are automatically penalized — the same way a substation outage propagates through a grid.

Operator control

Manual override lets dispatchers pin a catastrophic score (e.g., confirmed incident) for 30 s and immediately trigger the cascade — bypassing the stream's natural detection lag.

Streaming by default

Every score is recomputed from a 5-second tumbling window over fresh sensor data. No nightly batch; no stale snapshots.

04 · Urban Health Score

One weighted score per district, every 5 seconds.

05 · Cascade Model

Failures propagate through an adjacency graph.

  • When a district's score drops ≥ 20 points in a single 5-second window, the worker publishes a ripple to city.cascade.
  • Every adjacent district takes a −12 point penalty, written as a source=cascade row in history.
  • Manual catastrophes (POST /add with catastrophic:true) publish the same ripple — so operator-confirmed incidents propagate immediately.
  • Adjacency is defined per-district in config.yaml — roughly geographic, 2–4 neighbors per seed.
Drop Downtown and 10 neighboring districts get hit in the same tick — Mass Ave, Fountain Square, Near Eastside, Near Westside, Riverside, Crown Hill …

07 · Control Plane

Operators can pause, resume, and reset the pipeline.

Pause

POST /pipeline/pause — records keep arriving from Kafka (offsets advance, no consumer lag), but nothing is buffered, scored, or cascaded. Onboarded climbs; processed stays flat.

Start

POST /pipeline/start — worker resumes normal aggregation within ~400 ms of the click.

Reset counters

POST /pipeline/reset — zeros the session record counters so demos and load tests start from a clean baseline.

Why it matters

A single pane of glass is only useful if operators trust what they're seeing — including the ability to stop feeding garbage data into it mid-incident.

Alerting
